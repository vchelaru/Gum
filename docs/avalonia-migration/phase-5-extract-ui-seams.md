# Phase 5 — Extract UI-framework seams

## Purpose

Make the existing WPF Gum tool depend on UI-neutral interfaces at every UI boundary, with a WPF implementation of each seam wired in first so there is **zero behavior change**. This proves each abstraction inside the shipping WPF build before any Avalonia code exists. The phase also splits the single `AddGum()` composition root into a shared, UI-neutral `AddGumCore()` plus a per-head `AddGumWpf()` (and a stubbed `AddGumAvalonia()` for later), pushing every WPF-typed registration into the WPF-specific extension.

The success criterion is concrete and binary: after this phase, `dotnet build GumFull.sln` succeeds and the WPF tool behaves identically to before — same tabs, dialogs, theming, property grid, and plugin registration.

## Decisions & rationale

- **Decision:** Abstraction-first, but the WPF implementation lands first with zero behavior change. **Reason:** Defining a seam and immediately wiring a WPF impl behind it proves the abstraction inside the shipping build before any Avalonia code exists, so a defective contract surfaces as a compile/behavior failure in a build we can actually run today. **Direction:** Success is binary — `dotnet build GumFull.sln` is green and the WPF tool behaves identically (same tabs, dialogs, theming, property grid, plugin registration). Treat any observable delta as a regression, not a "fix."
- **Decision:** Split the single composition root into `AddGumCore()` (UI-neutral) + `AddGumWpf()` (WPF head) + a stub `AddGumAvalonia()`, with `AddGumCore()` and its helpers moved to a neutral shared assembly. **Reason:** This shape lets the compiler and the DI container enforce head purity — `AddGumCore()` becomes the single, audited "no `System.Windows`" surface, and the Avalonia head can compose the exact same core services WITHOUT referencing the WPF `Gum` project. It also matches the repo's existing constructor-injection direction. **Direction:** Move `AddGumCore()` (and the generic `ForEachConcreteTypeAssignableTo` helper) into the neutral shared assembly (`Gum.UiAbstractions` if it stays dependency-light, else a small `Gum.Composition` project); keep `AddGumWpf()` in `Gum`. Change `file static` → `public`/`internal` as needed. The purity guarantee is enforced by a real test (see checklist), not a grep.
- **Decision (must-fix — `AddGumCore()` cannot scan the head assembly):** The `ViewModel` transient scan (`Builder.cs:78-81`, `ForEachConcreteTypeAssignableTo<ViewModel>(typeof(GumBuilder).Assembly, ...)`) and `AddViewModelFuncFactories(... typeof(ServiceCollectionExtensions).Assembly)` (`Builder.cs:182`) both scan the **WPF head assembly** and register WPF-typed view-models — `MainPanelViewModel`, `PluginTab`, and the synthesized `Func<FrameworkElement, PluginTab>` factory. If those two registrations run inside `AddGumCore()`, the neutral core registers WPF types and **fails the purity test**. **Reason:** the scan is assembly-driven, and the only assembly with concrete VMs today is the WPF `Gum` project; the generic *mechanism* is neutral but the *assembly it is pointed at* is not. **Direction:** the VM transient scan and `AddViewModelFuncFactories` must NOT be hard-coded into the neutral core. Either (a) have each head call a head-specific registration step that passes its own assembly in (the WPF head scans the `Gum` assembly from `AddGumWpf()`, the Avalonia head scans its own from `AddGumAvalonia()`), or (b) move the scan calls themselves into `AddGumWpf()`/`AddGumAvalonia()` while the generic `ForEachConcreteTypeAssignableTo` / func-factory *helpers* live in the shared assembly. Pick (a) or (b) during implementation, but the call that names an assembly is head-owned, not core. Until VMs are de-WPF'd (see the ViewModel sweep decision below) the scanned types are not portable anyway.
- **Decision (ViewModel de-WPF sweep is real Phase 5 work):** ~20 concrete view-models expose WPF types on public members and must be swept while the WPF head stays green. The VM base (`Gum.Mvvm.ViewModel`) is already portable — the leak is in the concrete VMs: `System.Windows.Visibility` (via `.ToVisibility()` in `BehaviorsViewModel`, `AlignmentViewModel`, `ImportFromGumxViewModel`, and the VariableGrid `MainControlViewModel`, among others), `System.Windows.Media.Imaging.BitmapImage` (`RecentItemViewModel`, `SearchItemViewModel`), `System.Windows.Data.ICollectionView` (`ImportFromGumxViewModel`, `ImportFileViewModel`), `System.Windows.Threading.DispatcherTimer` (`PerformanceViewModel`), and `Microsoft.Xaml.Behaviors.Core` in `DialogViewModel`. **Reason:** these VMs are picked up by the transient scan and are intended to be shared by both heads, so leaving WPF on their public surface either fails purity or forces the Avalonia head to reference WPF. **Direction:** sweep each leak with the WPF head staying byte-for-byte green — replace `Visibility` properties with `bool` and retain a WPF `BoolToVisibility` converter in the WPF head; replace `BitmapImage`/`ICollectionView`/`DispatcherTimer` with neutral equivalents (image-source handle, `IFilteredTabView`/neutral observable, the existing `PeriodicUiTimer` or an `IDispatcher`-driven timer); excise `Microsoft.Xaml.Behaviors` from `DialogViewModel`.
- **Decision:** Plugin tab/panel content flows through a neutral `IPanelContent` handle, and `MainPanelViewModel` is the **WPF implementation of `ITabManager`** — registered ONLY in `AddGumWpf()`, not reused as-is by Avalonia. **Reason:** `MainPanelViewModel` holds six `System.Windows.Data.ICollectionView` collections, performs `WindowsFormsHost` wrapping, and materializes `FrameworkElement` — all of which are WPF-only. Naming it the WPF `ITabManager` impl (rather than a shared component) keeps those types out of the neutral surface. **Direction:** Neutralize `PluginTab.Content` / `CustomHeaderContent` and the `Func<FrameworkElement, PluginTab>` factory to the `IPanelContent` handle; extract a neutral base view-model (column widths, the `IPanelContent`-typed tab model, teardown / layout persistence) shared by both heads, with a thin per-head subclass owning the framework-specific bits. The Avalonia head gets its OWN `ITabManager` implementation (Phase 10).
- **Decision (`PluginTab` and `TabLocation` relocate with `ITabManager`):** `ITabManager.AddControl(...)` *returns* `PluginTab`, so moving `ITabManager` into `Gum.UiAbstractions` forces `PluginTab` and `TabLocation` to move with it (it is not enough to merely retype `PluginTab.Content`). **Reason:** the neutral assembly cannot reference a type that lives in the WPF `Gum` project, and `PluginTab` is the return type on the neutral contract. **Direction:** relocate `PluginTab` and `TabLocation` to `Gum.UiAbstractions` as part of this phase. Both are tractable: `TabLocation` is a plain enum, and `PluginTab` already derives from the portable `Gum.Mvvm.ViewModel` and uses `IMessenger` — so the move drags `Gum.Mvvm.ViewModel`/`IMessenger` reachability into (or alongside) `Gum.UiAbstractions`. Confirm `Gum.Mvvm.ViewModel` and `CommunityToolkit.Mvvm`'s `IMessenger` are referenceable from the neutral assembly (they are framework-neutral); if `Gum.UiAbstractions` must stay dependency-light, route the move through the small `Gum.Composition`/shared MVVM assembly instead. This is an explicit relocation task, not a one-line property retype.
- **Decision:** Property grid is contract-only this phase (`IPropertyGrid` + `IDataUi`); the Avalonia impl is Phase 20. **Reason:** Defining the contract while only the WPF impl exists means a contract defect surfaces in the green WPF build, before any second implementation can mask it. **Direction:** Express the contract over the data-driven member model and wrap the existing `DataUiGrid`/`MainPropertyGrid` so the Variables tab is byte-for-byte unchanged.
- **Decision (must-fix — `IPropertyGrid` cannot be both WPF-free AND in `Gum.UiAbstractions` this phase):** The member model the contract is expressed over is **not** WPF-free: `WpfDataUi.DataTypes.MemberCategory` exposes public `System.Windows.Media.Brush? HeaderColor` (`MemberCategory.cs:15`) and `System.Windows.Visibility Visibility` (`MemberCategory.cs:17`), and `WpfDataUi.DataTypes.InstanceMember` exposes `Dictionary<string, RoutedEventHandler> ContextMenuEvents` (`InstanceMember.cs:58`). A contract typed over those members therefore **cannot** be WPF-free, so it cannot live in the neutral `Gum.UiAbstractions` this phase without dragging WPF in. **Reason:** relocating/mirroring the member model to strip those types is explicitly a Phase 20 decision (the member-model relocate-vs-mirror call is deferred), so the neutral types do not exist yet. **Resolution (chosen):** for Phase 5, **drop the "`IPropertyGrid` exposes no WPF types / lives in `Gum.UiAbstractions`" claim.** Keep the `IPropertyGrid`/`IDataUi` contracts in a **WPF-tolerant location** (the `Gum` head, or `WpfDataUi` itself) until Phase 20 relocates the member model and a neutral mirror exists. The Phase 5 deliverable is narrowed to: `PropertyGridManager` talks to `IPropertyGrid` instead of `DataUiGrid` directly, and `IDataUi` is named as the canonical per-editor seam — *not* a WPF-free contract in the neutral assembly. (The alternative — minting a neutral mirror of the member model in this phase — is rejected here to avoid pulling Phase 20's relocate-vs-mirror decision forward.)

## Scope

### In scope

- Defining UI-neutral seam interfaces (Dispatcher, Dialogs, Theming, Panel/tab host, plus the OS-integration seams: save/folder pickers, clipboard, reveal-in-file-manager) in the `Gum.UiAbstractions` project (`Gum.UiAbstractions\`, at the repo root) created in Phase 1. **Exception:** the `IPropertyGrid`/`IDataUi` property-grid contracts are defined this phase but kept in a **WPF-tolerant location** (not `Gum.UiAbstractions`), because the member model they are typed over carries WPF on its public surface — see the property-grid Decision and Task 5.
- Relocating `PluginTab` and `TabLocation` into the neutral assembly (they are the return type of the neutralized `ITabManager`).
- Providing a **WPF implementation** of each new/changed seam and registering it through DI.
- Removing UI-framework type leakage from the interfaces themselves: `System.Windows.Window` from `IDialogService`, `System.Windows.FrameworkElement` / `System.Windows.Forms.Control` from `ITabManager` and `PluginTab`.
- **Sweeping WPF types off the public surface of the ~20 concrete view-models** picked up by the transient scan (Visibility→bool + retained WPF converter, plus replacing `BitmapImage`/`ICollectionView`/`DispatcherTimer`/`Microsoft.Xaml.Behaviors`) so the scanned VM set is portable (Task 7).
- Removing the stray `Application.Current.Dispatcher` reference in `Builder.cs` by routing it through a WPF-head factory.
- Splitting `Builder.cs` `AddGum()` into `AddGumCore()` + `AddGumWpf()` (and a placeholder `AddGumAvalonia()`), with the **head-owned** VM transient scan + func-factory registration (they name the head assembly, so they cannot sit in the neutral core), and updating `GumBuilder.CreateHostBuilder` to call the WPF head extension.

### Out of scope

- Any Avalonia control, view, or window. `AddGumAvalonia()` is a stub/placeholder only.
- Re-implementing the Variables property grid on a new framework — this phase only **defines** the `IPropertyGrid` contract; the full per-head implementation is Phase 20.
- Hosting the XNA editor canvas on a new framework (Phase 15) and the Avalonia shell skeleton (Phase 10).
- Changing plugin authoring APIs in a breaking way. The public plugin tab entry points `PluginBase.CreateTab(...)` (`Gum\Plugins\BaseClasses\PluginBase.cs:346`) and `PluginBase.AddControl(...)` (`:354`) — both currently typed `System.Windows.FrameworkElement` — and all internal-plugin call sites must keep working.
- Theming visual redesign — only the contract/implementation boundary is touched.

### 4. Panel / tab host seam

This is the largest change — it is where the UI-framework types actually leak through an interface.

- `Gum\Managers\ITabManager.cs` leaks both `System.Windows.Forms.Control` and `System.Windows.FrameworkElement`:
  ```
  PluginTab AddControl(System.Windows.Forms.Control control, string tabTitle, TabLocation tabLocation);
  PluginTab AddControl(FrameworkElement element, string tabTitle, TabLocation tabLocation = TabLocation.CenterBottom);
  ```
- `Gum\ViewModels\MainPanelViewModel.cs` (implements `ITabManager`) leaks the same types. The WinForms `AddControl` overload wraps the control in a `WindowsFormsHost` (line 108). The `FrameworkElement` overload (lines 110–124) contains the airspace hack that sets `host.Margin` when the element is itself a `WindowsFormsHost` **and** the tab title is `"Editor"` (lines 116–120) — see the note below about this branch.
- `Gum\Plugins\PluginTab.cs` exposes `FrameworkElement Content` (line 29, read-only) and `FrameworkElement? CustomHeaderContent` (line 35), takes a `FrameworkElement` in its ctor (line 85), and derives from the neutral `Gum.Mvvm.ViewModel`. `MainPanelViewModel` builds tabs through a `Func<FrameworkElement, PluginTab>` factory (`MainPanelViewModel.cs:43` field, `:59` ctor param). That factory is synthesized by `AddViewModelFuncFactories` (`Builder.cs:182`) because `PluginTab` is a `ViewModel` picked up by the transient scan.

**Approach:** introduce a neutral "panel content handle" so plugins register content without naming a UI framework.

- **Define `IPanelContent` as ONE concrete shape — a handle carrying a kind discriminator.** Resolve the "opaque handle vs `object Content` + discriminator" ambiguity now: pick the discriminated handle, because the two `AddControl` overloads otherwise collapse to identical neutral signatures (both would be `AddControl(IPanelContent, string, TabLocation)`, an illegal overload pair). Minimal surface:
  ```csharp
  public enum PanelContentKind { NativeControl, ViewObject }
  public interface IPanelContent
  {
      PanelContentKind Kind { get; }   // how the head should materialize Payload
      object Payload { get; }          // WinForms Control or framework-neutral view object; head casts per Kind
  }
  ```
  The head materializes `Payload` into a real control based on `Kind` (WPF head: `Kind == NativeControl` → wrap in `WindowsFormsHost`; `Kind == ViewObject` → cast to `FrameworkElement`). The discriminator lets a single `AddControl(IPanelContent content, string tabTitle, TabLocation tabLocation)` replace both leaking overloads without an ambiguous overload set, while the WPF head keeps its two strongly-typed `AddControl` entry points (below) as thin wrappers that build the right `IPanelContent`.
- Re-declare the neutral `ITabManager.AddControl(...)` over the single `IPanelContent` handle, keeping `TabLocation` and the `PluginTab` return type. The WPF head retains its `AddControl(System.Windows.Forms.Control, ...)` and `AddControl(FrameworkElement, ...)` overloads (Task 4 call-site list) as wrappers that construct `IPanelContent` with `Kind = NativeControl` / `ViewObject` respectively, so every existing call site compiles unchanged.
- Change `PluginTab.Content` / `CustomHeaderContent` to `IPanelContent` and update the factory from `Func<FrameworkElement, PluginTab>` to one over `IPanelContent`.
- **Relocate `PluginTab` and `TabLocation` into the neutral assembly.** Because `ITabManager.AddControl(...)` *returns* `PluginTab`, moving `ITabManager` into `Gum.UiAbstractions` requires `PluginTab` and `TabLocation` to move too — retyping `PluginTab.Content` alone is insufficient. `TabLocation` is a plain enum; `PluginTab` derives from the portable `Gum.Mvvm.ViewModel` and uses `IMessenger`, so the move drags `Gum.Mvvm.ViewModel`/`IMessenger` reachability into the neutral assembly (both are framework-neutral). If `Gum.UiAbstractions` must stay dependency-light, place `PluginTab`/`TabLocation` (and the MVVM base) in the shared `Gum.Composition` assembly instead. Verify the public plugin surface (`PluginBase.CreateTab`/`AddControl`, which return `PluginTab`) still compiles after the move.
- **`MainPanelViewModel` is the WPF implementation of `ITabManager` — it is NOT reused unchanged by Avalonia.** It holds six `System.Windows.Data.ICollectionView` collections, does `WindowsFormsHost` wrapping, and materializes `FrameworkElement`, so it is registered **only** in `AddGumWpf()`. The decision (locked with Phase 10): extract a **neutral base view-model** — column widths, the `IPanelContent`-typed tab model, teardown / layout persistence — shared by both heads, plus a **thin per-head subclass** owning the framework-specific bits. The WPF subclass converts the neutral handle into a `FrameworkElement` / `WindowsFormsHost` and keeps the `ICollectionView` plumbing (`CenterView`, `RightBottomView`, `RightTopView`, `CenterTopView`, `CenterBottomView`, `LeftView`); the Avalonia head gets its **own** `ITabManager` subclass (Phase 10). The `MainPanelControl.xaml` binding to `PluginTab.Content` must still resolve to a `FrameworkElement` in the WPF head — do this via a converter/template, not by leaking WPF onto the interface.
- The airspace hack (the `WindowsFormsHost`+`tabTitle == "Editor"` branch, lines 110–124) stays inside the WPF impl as a WPF-only detail. Document that it disappears entirely under Avalonia (no WinForms interop / airspace problem there). (See the airspace note in Risks: this branch does not actually fire for the real Editor tab, which passes a `Grid`.)
- **`ICollectionView` → tab-view adaptation ownership.** `MainPanelViewModel` exposes its six tab collections as WPF `ICollectionView` (`CenterView`/`RightBottomView`/`RightTopView`/`CenterTopView`/`CenterBottomView`/`LeftView`). `System.Windows.Data.ICollectionView` is a Windows-only WPF type, so the Avalonia head **cannot** bind it directly — doing so would pull a WPF assembly into the Avalonia graph and fail the Phase 7 purity guard. **A neutral filtered-tab-view seam is therefore mandatory, not optional** (locked with Phase 10). It is part of *this* phase's panel/tab-host seam family: a neutral `IFilteredTabView` (an observable, filtered, selection-tracking collection over `PluginTab`) lives in `Gum.UiAbstractions` alongside the `IPanelContent` handle and is exposed by the shared neutral base view-model; the WPF subclass adapts it over `ListCollectionView`, the Avalonia head supplies a native impl. Phase 10 consumes it; Phase 5 owns its definition. Minimal surface (implementer may extend):
  ```csharp
  public interface IFilteredTabView : IEnumerable<PluginTab>, INotifyCollectionChanged
  {
      PluginTab? SelectedTab { get; set; }   // selection tracking (replaces ICollectionView.CurrentItem)
      void Refresh();                          // re-evaluate the filter (replaces ICollectionView.Refresh)
  }
  ```
  The six WPF `ICollectionView` members (`CenterView`/`RightBottomView`/`RightTopView`/`CenterTopView`/`CenterBottomView`/`LeftView`) each become an `IFilteredTabView` on the neutral base view-model; the WPF subclass backs each with a `ListCollectionView` whose `Filter` selects tabs by `TabLocation`.
- **Do not break plugin registration.** Verify all `_tabManager.AddControl(...)` / `PluginBase` call sites still compile and behave identically. The complete set in this repo (verified by grep):
  - Public plugin surface: `Gum\Plugins\BaseClasses\PluginBase.cs` — `CreateTab(...)` (`:346`, calls `_tabManager.AddControl` at `:348`) and `AddControl(...)` (`:354`, calls at `:356`). Both take `System.Windows.FrameworkElement` only; there is **no** WinForms overload on `PluginBase`.
  - Direct `_tabManager.AddControl(...)` internal call sites:
    - `Gum\Plugins\InternalPlugins\VariableGrid\PropertyGridManager.cs:170` ("Variables", passes the WPF `MainPropertyGrid`)
    - `Gum\Plugins\InternalPlugins\TreeView\ElementTreeViewManager.cs:765` ("Project")
    - `Gum\Plugins\InternalPlugins\Output\MainOutputPlugin.cs:19`, `Errors\MainErrorsPlugin.cs:81`, `StatePlugin\MainStatePlugin.cs:124`, `Behaviors\MainBehaviorsPlugin.cs:50`, `FileWatchPlugin\MainFileWatchPlugin.cs:45`, `ProjectPropertiesWindowPlugin\MainPropertiesWindowPlugin.cs:82`, `AlignmentButtons\AlignmentMainPlugin.cs:24`
    - `Gum\CodeOutputPlugin\MainCodeOutputPlugin.cs:485`, `Gum\StateAnimationPlugin\MainStateAnimationPlugin.cs:301`, `Gum\TextureCoordinateSelectionPlugin\Logic\ControlLogic.cs:139`
    - `Tool\EditorTabPlugin_XNA\MainEditorTabPlugin.cs:1166` — the `"Editor"` tab. **It passes a `System.Windows.Controls.Grid` (a `FrameworkElement`) that has a `WindowsFormsHost` nested *inside* it** (the host wraps `gumEditorPanel`, the XNA canvas; see `MainEditorTabPlugin.cs:1152-1166`). Because the argument is a `Grid`, not a bare `WindowsFormsHost`, the airspace branch in `MainPanelViewModel.AddControl` (line 116, `element is WindowsFormsHost host && tabTitle == "Editor"`) does **not** fire for this call — see the airspace note in Risks.
  - Sites that go through `PluginBase` (not directly through `_tabManager`): `Gum\PerformanceMeasurementPlugin\MainPlugin.cs:34` and `Gum\Plugins\InternalPlugins\Undos\MainPlugin.cs:22` both call `PluginBase.AddControl(FrameworkElement, ...)`.

  Internal sites pass either `System.Windows.Forms.Control` (the WinForms `AddControl` overload) or WPF `FrameworkElement`s. The WPF head's `AddControl` overloads must continue to accept those exact types (wrapping WinForms in `WindowsFormsHost`) so every call site compiles and behaves identically. Preserve the `PluginBase.CreateTab`/`AddControl` `FrameworkElement` signatures.

### 5. Property grid contract

This phase **defines the contract only**; the Avalonia implementation is Phase 20.

- The Variables tab is built in `Gum\Plugins\InternalPlugins\VariableGrid\PropertyGridManager.cs` (`mainControl = new Gum.MainPropertyGrid()` at line 168, then `_tabManager.AddControl(mainControl, "Variables", ...)` at line 170; `mVariablesDataGrid = mainControl.DataGrid` at line 172, typed `WpfDataUi.DataUiGrid` — field declared at line 48). The grid itself is the WpfDataUi `DataUiGrid` (`WpfDataUi\DataUiGrid.cs`), driven by a `MemberCategory` / `InstanceMember` member model (the category type is `WpfDataUi.DataTypes.MemberCategory` — there is no `DataUiCategory` type).
- The concrete surface `PropertyGridManager` uses on `DataUiGrid` (the contract `IPropertyGrid` must cover): `Instance` (set; `DataUiGrid.cs:73`), `Categories` (`BulkObservableCollection<MemberCategory>`; `:122`), `SetCategories(IList<MemberCategory>)` (`:189`), `SetMultipleCategoryLists(List<List<MemberCategory>>)` (`:665`), `Refresh()` (`:575`), and `IsEnabled`. See `PropertyGridManager.cs:306,380,401,405,423,476-483,504` for the call sites.
- Define an `IPropertyGrid` seam over this **data-driven member model**, not over the WPF control. The contract is expressed in member-model terms (set/replace the categorized member lists, set the bound instance, refresh, `IsEnabled`, expose change events) covering exactly the surface above.
- **Must-fix — `IPropertyGrid` is NOT WPF-free this phase, and does NOT live in `Gum.UiAbstractions` this phase.** The member model the contract is typed over carries WPF on its public surface: `MemberCategory.HeaderColor` is `System.Windows.Media.Brush?` (`MemberCategory.cs:15`), `MemberCategory.Visibility` is `System.Windows.Visibility` (`MemberCategory.cs:17`), and `InstanceMember.ContextMenuEvents` is `Dictionary<string, RoutedEventHandler>` (`InstanceMember.cs:58`). A contract over `MemberCategory`/`InstanceMember` therefore cannot be WPF-free. **Resolution (chosen):** keep `IPropertyGrid` and `IDataUi` in a **WPF-tolerant location** for Phase 5 — the `Gum` head, or `WpfDataUi` itself — NOT in `Gum.UiAbstractions`. The Phase 5 deliverable is the *boundary*, not neutrality: `PropertyGridManager` talks to `IPropertyGrid` instead of `DataUiGrid` directly. Minting a neutral mirror of the member model (which would let `IPropertyGrid` be WPF-free and live in the neutral assembly) is **deferred to Phase 20** alongside the member-model relocate-vs-mirror decision; do not pull it forward here. (The Avalonia `IPropertyGrid` impl is also Phase 20, so nothing in this phase needs the contract to be neutral yet.)
- **Per-editor `IDataUi` contract.** The grid drives each member editor through the `IDataUi` interface (`WpfDataUi\IDataUi.cs`), not the WPF base class. This phase establishes `IDataUi` as the canonical editor contract that the grid talks to, so the Avalonia editors Phase 20 authors implement the same contract. `IDataUi` stays in `WpfDataUi` (WPF-tolerant) for now; relocating or mirroring it to a neutral assembly is the Phase 20 decision — this phase only names it as the seam boundary so Phase 20's "Phase 5 equivalent" of `IDataUi` is owned and not orphaned.
- The WPF implementation of `IPropertyGrid` wraps the existing `DataUiGrid`/`MainPropertyGrid` so the Variables tab is byte-for-byte the same in this phase. **No visual or behavioral change.**
- Document that an Avalonia `IPropertyGrid` implementation (a new data-driven grid), Avalonia `IDataUi` editors, AND the neutral mirror/relocation of the member model (so `IPropertyGrid` can move to `Gum.UiAbstractions` WPF-free) are the Phase 20 deliverable; this phase just locks the boundary so `PropertyGridManager` no longer references `DataUiGrid` directly (or references it only through the WPF head).

### 6. OS-integration seams (save/folder pickers, clipboard, reveal-in-file-manager)

Several cross-platform OS-integration surfaces bypass the existing seams and call Windows-only APIs directly. Add a seam for each with a **WPF implementation now**; the Avalonia implementations are deferred (Phase 20 for save pickers that ride `IStorageProvider`, Phase 25 for the rest). All call sites below are verified.

- **Save / folder pickers.** `IDialogService` only exposes `OpenFile` — it has no save path. The tool constructs native save dialogs directly at:

### 7. ViewModel de-WPF sweep

The `ViewModel` transient scan registers concrete VMs that are meant to be shared by both heads, but ~20 of them expose WPF types on **public** members. These must be swept so the scanned VM set is portable — otherwise the core (or the Avalonia head) drags WPF in, and the purity test fails. The MVVM base itself (`Gum.Mvvm.ViewModel`) is already portable; the leaks are all in concrete VMs. Do the sweep with the WPF head staying byte-for-byte green.

Leaks to fix (verified by grep; the list below is representative, not exhaustive — sweep all concrete VMs):

- **`System.Windows.Visibility`** via `.ToVisibility()` and `Visibility`-typed properties, e.g. `BehaviorsViewModel`, `AlignmentViewModel`, `ImportFromGumxViewModel`, and the VariableGrid `MainControlViewModel`. **Fix:** expose `bool` on the VM and keep a WPF `BoolToVisibility` converter (the repo already has `Gum\Converters\BoolToVisibilityConverter.cs`) in the WPF head's XAML so the WPF view is unchanged.
- **`System.Windows.Media.Imaging.BitmapImage`** on `RecentItemViewModel`, `SearchItemViewModel`. **Fix:** replace with a neutral image-source handle (path/`Uri`/byte source); the WPF view materializes a `BitmapImage` from it.
- **`System.Windows.Data.ICollectionView`** on `ImportFromGumxViewModel`, `ImportFileViewModel`. **Fix:** replace with a neutral observable/filtered collection (the `IFilteredTabView` pattern from Task 4, or a plainer neutral observable where selection/filter isn't needed).
- **`System.Windows.Threading.DispatcherTimer`** on `PerformanceViewModel`. **Fix:** drive it off the existing neutral `PeriodicUiTimer` / `IDispatcher` instead of naming `DispatcherTimer` on the VM.
- **`Microsoft.Xaml.Behaviors.Core`** in `Gum\Services\Dialogs\DialogViewModel.cs`. **Fix:** excise the `Microsoft.Xaml.Behaviors` dependency from the VM (move any behavior/command glue into the WPF view or a WPF-head helper).

Each fix must leave the WPF build green with no observable behavior change (the WPF view keeps the same rendered result via converters/templates). Note this sweep is what makes the head-owned VM scan (Task 8) actually portable per head; until it lands, the scanned VM set is not neutral regardless of where the scan call lives.

### 8. DI composition split (`AddGumCore` / `AddGumWpf` / `AddGumAvalonia`)

- Split into:
  - `AddGumCore()` — everything UI-neutral: `Lazy<>`/`Lazier<>` (line 82), all logic/services/managers singletons, `IObjectFinder`/`PluginManager`/`IMessenger` (lines 86–88, 173), `IDialogViewResolver` (from `AddDialogs`, line 190), etc. Note: `IObjectFinder` stays a core singleton wrapping `ObjectFinder.Self` — that static singleton is intentional repo-wide and is not changed here.
  - **The `ViewModel` transient scan (lines 78–81) and `AddViewModelFuncFactories` (line 182) do NOT go in `AddGumCore()` (must-fix).** Both name the **WPF head assembly** (`typeof(GumBuilder).Assembly` / `typeof(ServiceCollectionExtensions).Assembly`) and register WPF-typed VMs (`MainPanelViewModel`, `PluginTab`, the `Func<FrameworkElement, PluginTab>` factory). - Decide where these extension methods live: they are currently in a `file static class ServiceCollectionExtensions` (line 73), which is invisible outside `Builder.cs`. If a separate Avalonia head assembly must call `AddGumCore()`, that scoping must change to `public`/`internal` in a shared location — verify accessibility during implementation. `ForEachConcreteTypeAssignableTo` and the func-factory helper (the other `file static` helpers, lines 202, 229) are core helpers and move with `AddGumCore`; the **calls** that point them at the head assembly stay in the head (above).

## Risks & notes

- **`MainPanelViewModel` is the WPF `ITabManager` impl, not a shared component.** Do not treat it as "reused as-is" by Avalonia. It carries six `System.Windows.Data.ICollectionView`, `WindowsFormsHost` wrapping, and `FrameworkElement` materialization, so it is registered only in `AddGumWpf()`. The resolution (locked with Phase 10) is a **neutral base view-model** (column widths, `IPanelContent`-typed tab model, teardown / layout persistence) plus a **thin per-head subclass**; the Avalonia head gets its own subclass in Phase 10. Any plan that registers `MainPanelViewModel` from the core or hands it to the Avalonia head unchanged is wrong.
- **Airspace hack disappears under Avalonia.** The `WindowsFormsHost` + `host.Margin` workaround in `MainPanelViewModel.cs:116-120` exists only because WPF/WinForms interop blocks mouse hits on the grid splitter and window resize handle. It must stay inside the WPF head and must not be promoted onto the neutral interface. Avalonia has no WinForms airspace problem.
- **The airspace branch may be partially dead — preserve behavior exactly, don't "fix" it.** The branch at `MainPanelViewModel.cs:116` only fires when the `element` argument *is itself* a `WindowsFormsHost` and the title is `"Editor"`. The actual Editor tab (`MainEditorTabPlugin.cs:1166`) passes a `Grid` with the `WindowsFormsHost` nested inside, so this branch does not run for it. Whatever the current behavior is, this phase must reproduce it identically — do not remove or "repair" the branch as part of the seam extraction; treat any behavior delta as a regression. Flag the apparent mismatch to the architect rather than changing it here.
- **Do not break plugin registration.** External and internal plugins call `_tabManager.AddControl(...)` directly (WinForms `Control` or WPF `FrameworkElement`) or via `PluginBase.CreateTab`/`AddControl` (`FrameworkElement` only). There are ~16 internal call sites (enumerated in Task 4) plus the public `PluginBase` surface. Keep WPF-head overloads that accept those exact types and wrap them so every call site compiles and behaves identically. Treat any change to the public plugin surface as a regression.
- **`PluginTab.Content` is data-bound in XAML.** `MainPanelControl.xaml` binds to `Content` as a `FrameworkElement`. When `Content` becomes a neutral handle, materialize it to a `FrameworkElement` via a value converter / DataTemplate in the WPF head rather than leaking the type back onto the model.
- **Several contracts are already neutral.** `IDispatcher`, `IDialogService`, `IThemingService`/`IEffectiveThemeSettings`, and `IUiSettingsService` already avoid WPF types on their interfaces; the work for those is mostly **relocation of registrations** into `AddGumWpf()`, not interface surgery. Don't over-engineer them.
- **`file static class` scoping.** `ServiceCollectionExtensions` (and the `ServiceCollectionHelpers` / `ViewModelFuncFactoryRegistration` helpers) in `Builder.cs` are all `file`-scoped, so nothing outside `Builder.cs` can call them. If a separate Avalonia head assembly must call `AddGumCore()`, that scoping must change to `public`/`internal` and likely move out of the WPF tool project. Decide deliberately during implementation.
- **`Application.Current.Dispatcher` is the only stray reference** in tool source (`Builder.cs:183`); the grep hit in `docs_projects\11_WpfDataUi.md` is documentation, not code. Removing the `Builder.cs` reference fully clears the dispatcher seam.
- **`AddGumCore()` cannot run the VM transient scan or the func-factory registration as currently bucketed (must-fix).** `Builder.cs:78-81` scans `typeof(GumBuilder).Assembly` (the WPF head) and `Builder.cs:182` calls `AddViewModelFuncFactories(typeof(ServiceCollectionExtensions).Assembly)` — both register WPF-typed VMs (`MainPanelViewModel`, `PluginTab`, the `Func<FrameworkElement, PluginTab>` factory). Leaving these in the neutral core makes the core register WPF types and FAILS the purity test. The assembly-naming call must be head-owned (pass the head assembly into a head step, or move the scan calls into `AddGumWpf()`/`AddGumAvalonia()`); only the generic helpers may live in the shared assembly. See Task 8.
- **Property grid is contract-only this phase, and the contract is NOT WPF-free (must-fix coherence).** Resist implementing a new grid. The member model (`MemberCategory`/`InstanceMember`) carries WPF on its public surface (`Brush HeaderColor`, `Visibility Visibility`, `Dictionary<string, RoutedEventHandler> ContextMenuEvents`), so `IPropertyGrid`/`IDataUi` are defined this phase but kept in a **WPF-tolerant location** (the `Gum` head or `WpfDataUi`), NOT `Gum.UiAbstractions`. Do not claim `IPropertyGrid` is WPF-free or lives in the neutral assembly this phase — that is the Phase 20 deliverable (member-model mirror/relocation). The WPF impl wraps the existing `DataUiGrid` so the Variables tab is unchanged.
- **ViewModel de-WPF sweep is required, not cosmetic (Task 7).** ~20 concrete VMs expose `System.Windows.Visibility`, `BitmapImage`, `ICollectionView`, `DispatcherTimer`, or `Microsoft.Xaml.Behaviors` on public members. Because these VMs are scanned and meant to be shared, the sweep is what makes the head-owned scan actually portable. Every fix must keep the WPF build byte-for-byte green (Visibility→bool + retained WPF converter, etc.). The MVVM base `Gum.Mvvm.ViewModel` is already portable — do not "fix" it.
- **`CompositeMemberLogic.cs` is not fully neutralized by the clipboard seam.** It has a file-level `using System.Windows.Forms;` (`:13`), so routing its clipboard call through `IClipboardService` removes one WinForms reference but leaves the `using` and any other WinForms usage. Full neutralization of this file is out of scope for the clipboard seam — flag the residual and track it separately.
