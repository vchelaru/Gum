# Phase 1 — Scaffolding

## Purpose
Lay down the empty project skeleton for the Avalonia migration without changing any existing behavior. This phase only creates new, unreferenced projects (plus the planning folder) so later phases have somewhere to put code. No existing project is edited, no service is moved, and the shipping WPF + WinForms tool (`GumFull.sln`) must build and run exactly as before.

## Decisions & rationale
- **Both new projects target `net8.0` (single-target, not multi-target).** Reason: it matches the sibling headless projects `Gum.ProjectServices` and `GumCommon`, and `net8.0` is referenceable by both heads — the existing `net8.0-windows` WPF tool and the new `net8.0` Avalonia head can each consume a `net8.0` assembly. Direction: keep both csprojs single-target `net8.0`; only introduce a `netstandard2.0;net8.0` multi-target if a concrete non-net8 consumer ever appears (none does today).
- **Scaffold `Gum.Avalonia` from the official Avalonia template, do NOT hand-author the csproj/bootstrap.** Reason: hand-authoring is the root cause of the dry-run failures — it silently omits required pieces (`app.manifest`, `<AvaloniaResource Include="Assets\**"/>`, `BuiltInComInteropSupport`, the `<FluentTheme/>` resource wiring in `App.axaml`, and the `.UsePlatformDetect().WithInterFont()` builder calls) and leaves the Avalonia version unfilled. The `dotnet new avalonia.app` template produces a known-good, launchable shell with all of these wired and a concrete pinned 11.x version. Direction: install the templates (`dotnet new install Avalonia.Templates`), scaffold with `dotnet new avalonia.app -o Gum.Avalonia`, then apply the migration constraints (`net8.0`, `OutputType=WinExe`, no `UseWPF`/`UseWindowsForms`) as post-scaffold checks rather than building the project from a blank file.
- **`Gum.Avalonia` uses `OutputType=WinExe` with NO `UseWPF`/`UseWindowsForms`.** Reason: `WinExe` only suppresses the console window and is the correct, cross-platform output type for an Avalonia desktop app on all OSes; setting `UseWPF`/`UseWindowsForms` (or a `net8.0-windows` TFM) would pull in Windows-only frameworks and break macOS/Linux builds. Direction: the template emits `OutputType=WinExe` and `net8.0` already — verify these as post-scaffold checks and keep neither WPF nor WinForms enabled.
- **Pin Avalonia to the exact version the template resolves (current 11.x).** Reason: a single pinned version avoids mismatched-version runtime errors across the Avalonia package set, and every head and later phase assumes this one pin. The template pins a concrete 11.x patch at scaffold time; that resolved version is the pin. Direction: do not invent a version — take whatever exact version `dotnet new avalonia.app` resolves, confirm all Avalonia `PackageReference`s use that one version, and record it in the "Resolved Avalonia version" note below. **Resolved Avalonia version: _(record here once scaffolded — e.g. 11.x.y)_.**
- **`Gum.Avalonia` head namespace/`RootNamespace` is `Gum.Avalonia`.** Reason: it matches the project/folder name and the existing head-naming convention (the WPF head's root is `Gum`). Direction: set/confirm `RootNamespace` to `Gum.Avalonia` and use the `Gum.Avalonia` namespace in the scaffolded `App`/`MainWindow`/`Program` (the template defaults the namespace to the `-o` name, so this is a confirmation step, not a rewrite).
- **Solution folder: reuse the existing `Tools` folder, not a new `Avalonia` folder.** Reason: both new projects are head/seam tooling alongside `Gum.ProjectServices` and `Gum.Cli`, and reusing the existing folder avoids solution-structure churn. Direction: add both projects to the `Tools` solution folder in `AllLibraries.sln`.
- **Project location: repo root.** Reason: placing `Gum.Avalonia\` and `Gum.UiAbstractions\` at the repo root parallels the existing root-level WPF head `Gum\`, keeping the two heads visually side by side. Direction: create both projects at the repo root; every phase doc references them there.
- **`Gum.UiAbstractions` references NEITHER WPF NOR Avalonia.** Reason: it is the single seam assembly both heads depend on, so it must be framework-neutral — that is exactly what lets the WPF tool and the Avalonia head share it. Direction: keep it free of `UseWPF`, WinForms, and all Avalonia packages; the Phase 7 purity guard enforces this invariant.

## Scope

### In scope
- The planning folder `docs\avalonia-migration\` and this set of phase documents.
- A new Avalonia desktop application project `Gum.Avalonia` (`net8.0`, cross-platform, no WPF/WinForms references), scaffolded from the official `dotnet new avalonia.app` template. Compiles and launches an empty window; references nothing from the existing tool yet.
- A new `Gum.UiAbstractions` class library created empty (hand-authored, since the template story is only needed for the Avalonia head). It is the future home of the UI seams (Dispatcher, dialogs, theming, panel/tab host, property-grid contracts) but receives no interfaces in this phase — that is Phase 5's job.
- Solution wiring: add both new projects to `AllLibraries.sln` (the cross-platform, WPF-free solution) via `dotnet sln add` so they build in CI and the IDE, while keeping `GumFull.sln`'s plugin post-build behavior intact.
- A WPF third-party UI dependency inventory: a list of the WPF-only third-party UI libraries the existing tool depends on, each with a per-library Avalonia replacement decision (deliverable only — no implementation), to feed the later UI-replacement phases.

### Out of scope
- Defining any seam interfaces, services, or contracts in `Gum.UiAbstractions` (Phase 5).
- Any Avalonia views, XAML, view-model wiring, DI composition (`AddGumCore()` / per-head extensions), or canvas hosting (Phases 5/10–20).
- Touching `Gum\Gum.csproj`, `Gum\Program.cs`, `Gum\Services\Builder.cs`, or any plugin project.
- Splitting the DI composition root or extracting anything from the existing tool.
- macOS/Linux build/run validation of actual editor functionality (the Phase 1 Avalonia head is an empty shell; Windows is sufficient for this phase).

## Tasks
Perform these in order. Each is additive.

1. **Branch.** Create a feature branch off `master` (e.g. `feature/avalonia-scaffolding`). All migration work lands on branches, not `master`.
2. **Confirm the planning folder.** `docs\avalonia-migration\` already exists and holds the phase docs. This matches the repo convention of topic subfolders under `docs\` (the folder already contains `cli\`, `code\`, `gum-elements\`, `gum-tool\`).
3. **Create `Gum.UiAbstractions`.** Add `Gum.UiAbstractions\Gum.UiAbstractions.csproj` at the repo root (location resolved — see "Project location" in Risks & notes). Make it a plain SDK-style class library with no UI framework references:
   - `<TargetFramework>net8.0</TargetFramework>` — the architectural reason is that this is the single seam assembly both heads must reference, so it has to sit on a TFM both can consume: the existing tool (`net8.0-windows`) and the new Avalonia head (`net8.0`) can each reference a `net8.0` assembly. It also matches the sibling `Gum.ProjectServices` and `GumCommon`. (A `netstandard2.0;net8.0` multi-target is possible if a future non-net8 consumer ever needs it, but no current consumer does — keep it `net8.0` unless a concrete need appears.)
   - `<Nullable>enable</Nullable>`, `<LangVersion>12.0</LangVersion>` to match the sibling projects (`Gum.ProjectServices`, `GumCommon`). `ImplicitUsings` defaults to disabled for class libraries (and `GumCommon` sets it to `disable` explicitly); leave it at the default or set `disable` explicitly to mirror `GumCommon`.
   - No `ProjectReference` and no `PackageReference` in this phase. Leave the project body empty — an SDK-style class library with zero source files compiles fine.
4. **Scaffold `Gum.Avalonia` from the Avalonia template.** Do not hand-write the csproj or bootstrap files — generate them so `app.manifest`, `<AvaloniaResource Include="Assets\**"/>`, `BuiltInComInteropSupport`, the `<FluentTheme/>` wiring, and `.UsePlatformDetect().WithInterFont()` come in correctly and a concrete version is pinned:
   - Install the templates: `dotnet new install Avalonia.Templates`.
   - From the repo root, scaffold: `dotnet new avalonia.app -o Gum.Avalonia`. This produces `Gum.Avalonia\Gum.Avalonia.csproj`, `Program.cs` (with `[STAThread] static Main` calling `BuildAvaloniaApp().StartWithClassicDesktopLifetime(args)` and a `BuildAvaloniaApp()` that calls `.UsePlatformDetect().WithInterFont().LogToTrace()`), `App.axaml`/`App.axaml.cs` (with `<FluentTheme/>` in `Application.Styles`), `MainWindow.axaml` (+ `.cs`), `app.manifest`, and the `Assets\` folder wired via `<AvaloniaResource>`. The template pins all Avalonia packages (`Avalonia`, `Avalonia.Desktop`, `Avalonia.Themes.Fluent`, `Avalonia.Fonts.Inter`, Debug-only `Avalonia.Diagnostics`) to one exact 11.x version.
   - **Record the resolved Avalonia version** in this doc (the "Resolved Avalonia version" note in Decisions & rationale and the matching checklist item). Confirm every Avalonia `PackageReference` uses that single version.
   - **Post-scaffold checks (constraints, not rewrites):** confirm `<TargetFramework>net8.0</TargetFramework>` (not `net8.0-windows`), `<OutputType>WinExe</OutputType>`, `<Nullable>enable</Nullable>`, and `<RootNamespace>Gum.Avalonia</RootNamespace>` (template defaults the namespace to the `-o` name; confirm `App`/`MainWindow`/`Program` use the `Gum.Avalonia` namespace). Set `<LangVersion>12.0</LangVersion>` to match siblings if the template did not. `<UseWPF>` and `<UseWindowsForms>` MUST NOT be set, and there must be no reference to `Gum\Gum.csproj` or any WPF/WinForms assembly. This is a self-contained entry point, separate from the existing `System.Windows.Application` entry point in `Gum\Program.cs` (`Main`/`MainAsync`).
   - Optionally add a `ProjectReference` to `Gum.UiAbstractions`. Harmless even though that project is empty, and it documents intent. Later phases require this reference.
5. **Wire into `AllLibraries.sln` with `dotnet sln add`.** Use `dotnet sln AllLibraries.sln add Gum.UiAbstractions\Gum.UiAbstractions.csproj Gum.Avalonia\Gum.Avalonia.csproj` rather than hand-editing the solution file. `dotnet sln add` auto-generates each project's GUID **and** the full set of per-configuration build rows — important because `AllLibraries.sln` has 9 solution configurations (including a `Release_No_Diagnostics` config plus x64/x86 variants). A 2-config project like these maps onto those rows automatically (e.g. `Release_No_Diagnostics|Any CPU` → the project's `Release|Any CPU`, and the x64/x86 platforms → the project's `Any CPU`); hand-editing those mappings is error-prone, so let the CLI do it. Move both projects into the existing `Tools` solution folder (alongside `Gum.ProjectServices`/`Gum.Cli`) — do not create a new `Avalonia` folder. Do NOT add `Gum.Avalonia` to `GumFull.sln`. `Gum.UiAbstractions` may optionally be added to `GumFull.sln` (it is framework-neutral) via `dotnet sln GumFull.sln add Gum.UiAbstractions\Gum.UiAbstractions.csproj`; if you do, see the plugin post-build caveat in Risks & notes first.
6. **Inventory WPF-only third-party UI dependencies.** Produce an inventory (deliverable only, no implementation) of the WPF-specific third-party UI libraries the existing tool depends on, each with a proposed Avalonia replacement decision (replace-with-X, drop, or build-native). At minimum cover: `MaterialDesignThemes`, `ControlzEx`, `Xceed.Wpf.Toolkit`, `Xceed.Wpf.DataGrid`, `AvalonDock`, and `Microsoft.Xaml.Behaviors`. This inventory feeds the later UI-replacement work (Phases 20/25); it is awareness/planning output for this phase, not a code change.
7. **Verify builds.** Build `GumFull.sln` and confirm it still produces the WPF tool and copies plugins exactly as before (no diff in plugin output). Build the new projects (via `AllLibraries.sln` or directly) and confirm `Gum.UiAbstractions` and `Gum.Avalonia` compile, and that `Gum.Avalonia` launches an empty window on Windows.

## Risks & notes
- **Project location (resolved: repo root).** Both new projects live at the repo root (`Gum.Avalonia\`, `Gum.UiAbstractions\`), paralleling the existing root-level WPF head `Gum\`. Phase 10's "Key files" section references them at the same location. This is settled — every phase doc uses the repo-root paths; there is no `Tools\`-vs-root decision left open.
- **`GumFull.sln` `$(SolutionDir)` plugin post-build caveat (from CLAUDE.md).** Tool plugin projects with post-build steps that copy DLLs into `$(SolutionDir)Gum\bin\$(ConfigurationName)\Plugins\...` are: `StateAnimationPlugin`, `PerformanceMeasurementPlugin`, `TextureCoordinateSelectionPlugin`, `EventOutputPlugin`, `SkiaPlugin` (`Gum\SvgPlugin\SkiaPlugin.csproj`), `CodeOutputPlugin`, `GumFormsPlugin`, and `ImportFromGumxPlugin` (8 projects, all under `Gum\`). `$(SolutionDir)` is only defined when building through the solution, so the existing tool must keep being built via `GumFull.sln`. When editing `GumFull.sln` to add `Gum.UiAbstractions`, do not reorder, remove, or alter plugin project entries or their configurations — only append the new project. After editing, re-verify plugin output is unchanged.
- **Keep WPF out of the cross-platform graph.** The Avalonia head must never reference `Gum\Gum.csproj` or any WPF/WinForms assembly, and must not target `net8.0-windows`. Mixing them would break macOS/Linux builds and defeat the migration. `Gum.UiAbstractions` must stay framework-neutral (no `UseWPF`, no Avalonia packages) so both heads can consume it.
- **Empty project is intentional.** `Gum.UiAbstractions` ships with zero types in Phase 1. An SDK-style class library with no source files builds successfully; reviewers should not expect interfaces yet.
- **Avalonia version pin.** Do not invent or guess a version. The `dotnet new avalonia.app` template resolves and pins a single current 11.x version across all Avalonia packages; that resolved version is the pin. Confirm the packages are uniform, then record the exact version in the "Resolved Avalonia version" note in Decisions & rationale and the matching checklist item. Mismatched versions across the Avalonia package set cause runtime errors, so this uniformity check matters.
- **Scaffold, don't hand-author the Avalonia head.** Hand-writing the csproj/bootstrap is what made the earlier draft non-executable: it dropped `app.manifest`, `<AvaloniaResource Include="Assets\**"/>`, `BuiltInComInteropSupport`, the `<FluentTheme/>` wiring, and `.WithInterFont()`, and left the version blank. Always go through `dotnet new avalonia.app` and then apply the migration constraints as edits/checks on the generated files.
- **WPF third-party UI dependencies need replacement decisions.** The existing tool pulls in WPF-only UI libraries (`MaterialDesignThemes`, `ControlzEx`, `Xceed.Wpf.Toolkit`, `Xceed.Wpf.DataGrid`, `AvalonDock`, `Microsoft.Xaml.Behaviors`) that have no automatic Avalonia equivalent. None are addressed in Phase 1; the inventory task (Task 6) only records them and a proposed replacement per library so the later UI-replacement phases (20/25) can act on a known list.
