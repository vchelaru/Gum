# Phase 10 — Avalonia shell skeleton

## Purpose

Stand up a runnable Avalonia application head for the Gum editor: a main window, a composition root, the column/splitter layout, dockable tabs, and menus/toolbars — all bound to the **same** shared services and ViewModels that the WPF head uses. The goal is a launchable shell with placeholder panels, proving that the Phase 1 scaffolding (`Gum.Avalonia`, `Gum.UiAbstractions`) and the Phase 5 DI split (`AddGumCore()` + per-head extensions) actually compose into a window.

This phase does **not** host the editor canvas (Phase 15) or the real property grid (Phase 20). Those panels are placeholders here.

## Decisions & rationale

- **Layout: plain Avalonia `TabControl` + `GridSplitter`, NOT Dock.Avalonia.** Reason: the verified WPF shell (`MainPanelControl.xaml`) is plain `TabControl`s in a `Grid`/`GridSplitter` — there is no docking framework in the live layout (AvalonDock appears only as a couple of utility types in `Dialog.cs`). Direction: reproduce that structure 1:1 with Avalonia's built-in `TabControl`/`GridSplitter`. Dock.Avalonia is a **future option only** if drag-rearrange/float/persisted-dock-layout is later wanted; it brings its own layout model/serialization and is out of scope here.
- **Neutral tab-view seam over `ICollectionView` is mandatory, not optional.** Reason: `System.Windows.Data.ICollectionView`/`ListCollectionView` are WPF types living in a Windows-only assembly; direct-binding them from the Avalonia head pulls a WPF assembly into that head's reference graph, which the Phase 7 purity guard rejects (`ICollectionView` is not reachable on a non-Windows TFM). Direction: introduce the neutral filtered-view seam in `Gum.UiAbstractions` (Task 6 option (b)) with per-head impls; the direct-bind shortcut (option (a)) is not viable.
- **`MainPanelViewModel` is NOT reused unchanged across heads.** Reason: it carries six `ICollectionView` properties, `WindowsFormsHost` wrapping, and `FrameworkElement` materialization — all WPF/WinForms. Direction: per Phase 5, `MainPanelViewModel` becomes the **WPF impl of `ITabManager`** (registered only in `AddGumWpf`); the Avalonia head gets its **own** `ITabManager` impl. A neutral base VM (column widths, `IPanelContent`-typed tab model, teardown/layout-persistence) is shared, with a thin per-head subclass on top.
- **Menus: static AXAML menu, do not port `MenuStripManager` yet.** Reason: the skeleton only needs to prove command-service binding works end to end; the full imperative menu model is a later concern. Direction: hand-author a small representative `Menu` in AXAML wired to the command services; flag a head-agnostic menu model to the architect for a later phase.
- **Window chrome: standard window decoration now, custom chrome deferred to Phase 25.** Reason: the WPF window's ControlzEx `WindowChromeWindow` (hand-built caption buttons + hit-test regions) is non-trivial to reproduce and unrelated to proving the shell composes. Direction: use a standard Avalonia window decoration this phase; revisit chrome parity in Phase 25.

## Scope

### In scope

- An Avalonia `Application` subclass + `BuildAvaloniaApp()` builder and a desktop entry point using `StartWithClassicDesktopLifetime`.
- A composition root for the Avalonia head that calls `AddGumCore()` + `AddGumAvalonia()`, mirroring the host/`IMessenger`/`Locator` bootstrapping in `Gum/Program.cs`.
- An Avalonia main window (AXAML) replicating the column/splitter layout currently produced by `MainPanelControl.xaml`, bound to the Avalonia head's own `ITabManager`/neutral panel VM (a thin subclass over the shared neutral base VM — **not** `MainPanelViewModel`, which is the WPF `ITabManager` impl; see Decisions & rationale).
- Tab regions for the six panel locations. The current WPF shell renders these with plain `TabControl`s inside a `Grid`/`GridSplitter` layout (not a docking framework — see Key files), so plain Avalonia `TabControl` + `GridSplitter` is sufficient for parity. A full docking framework (Dock.Avalonia) is **optional** and only needed if drag-rearrange/float/persisted-dock-layout is later desired (deferred; see Risks).
- Menus and toolbars re-authored in AXAML, bound to the existing command services (`IEditCommands`, `IFileCommands`, `IGuiCommands`, `IWireframeCommands`).
- App lifecycle parity: send `ApplicationStartupMessage` on startup and drive `ApplicationTeardownMessage` (collect + run teardown actions) on exit.
- The neutral tab-view seam that replaces `MainPanelViewModel`'s `ICollectionView` tab collections (mandatory — `ICollectionView` is WPF-only and not reachable from the Avalonia head; see Decisions & rationale and Risks).
- Placeholder content for the five active tab regions (Left/CenterTop/CenterBottom/RightTop/RightBottom). `TabLocation.Center` is `[Obsolete]` and `CenterView` is unused by the live layout — do not wire it.

### Out of scope

- Editor canvas / wireframe rendering host (Phase 15).
- Real property grid and editor-detail views (Phase 20).
- A full docking framework (Dock.Avalonia drag/float/dock-layout serialization) — the current shell does not use one for its layout; defer to a later phase if ever needed.
- Plugin tab content beyond placeholders, theming polish, and final visual styling (Phase 25).
- Custom window chrome / caption-button parity with the ControlzEx `WindowChromeWindow` (the current WPF title bar in `MainWindow.xaml`). Use a standard Avalonia window decoration for this phase; chrome can be revisited in Phase 25.
- Replacing `ObjectFinder.Self` or other intentionally-static singletons.

## Tasks

1. **Avalonia App + builder.** Create the `Application` subclass in `Gum.Avalonia` with `Initialize()` loading app-level AXAML resources, and a `BuildAvaloniaApp()` that wires `UsePlatformDetect()`. Add a `Main` entry point (`[STAThread]`) that builds the generic host (reusing the shared `GumBuilder.CreateHostBuilder(args)` if it is exposed/made accessible by the Phase 5 split, otherwise mirroring it), registers it with `Locator`, starts it, then calls `BuildAvaloniaApp().StartWithClassicDesktopLifetime(args)`. Mirror the structure of `Gum/Program.cs:Main`/`MainAsync`. Note the WPF entry point sets WPF/WinForms-specific render/visual-style options (`Program.cs` lines 37-38) — those are WPF-only and must **not** be copied into the Avalonia head.

2. **Composition root.** In the Avalonia head, build the host via the Phase 5 split: `services.AddGumCore()` (shared, head-agnostic registrations carved out of today's `AddGum` in `Gum/Services/Builder.cs`) plus `services.AddGumAvalonia()` (Avalonia window, Avalonia `IDispatcher`, Avalonia UI-seam impls for dialogs/theming/panel host/property-grid). Resolve the Avalonia main window from DI and assign it as `desktop.MainWindow` inside `OnFrameworkInitializationCompleted`.

3. **Window/`IDispatcher` registrations.** `Builder.cs` currently registers `IDispatcher` as `new AppDispatcher(() => Application.Current.Dispatcher)` (WPF) and `MainWindow`/`MainWindowViewModel` as singletons (lines ~177-183). These are WPF-specific and must move into `AddGumWpf`; `AddGumAvalonia` provides an Avalonia-backed `IDispatcher` (over `Avalonia.Threading.Dispatcher.UIThread`) and the Avalonia main window. `MainWindowViewModel` and the WPF `ITabManager` impl (`MainPanelViewModel`) are per-head and register only in `AddGumWpf`; `AddGumAvalonia` registers the Avalonia `ITabManager` impl. The shared neutral panel base VM (if registered) is the only piece that belongs in core (see the neutral tab-view seam in Task 6).

4. **Column/splitter layout (AXAML view only).** Author an Avalonia `UserControl`/`Window` that reproduces `MainPanelControl.xaml`:
   - Outer 5-column grid: `LeftColumn` (bound to `LeftColumnWidth`), `Auto` splitter, `MiddleColumn` (bound to `CenterColumnWidth`), `Auto` splitter, `RightColumn` (`*`).
   - Center grid (column 2): rows `1*` / `Auto` splitter / `2*` hosting `CenterTopView` and `CenterBottomView`.
   - Right grid (column 4): rows `*` / `Auto` splitter / bound `BottomRightHeight` hosting `RightTopView` and `RightBottomView`.
   - Left tab control bound to `LeftView`.
   Use Avalonia `GridSplitter` (built in). `MainPanelViewModel` exposes plain `double` widths (`LeftColumnWidth`/`CenterColumnWidth`/`BottomRightHeight`), so binding column/row widths needs a `double`↔`GridLength` converter equivalent to the WPF `GridLengthToDoubleConverter` / `GridDefinitionSync` behavior. Re-author or port that converter for Avalonia.

5. **Tab regions.** Replicate the WPF shell's tab regions. The WPF `MainPanelControl.xaml` uses plain `TabControl`s (one per region) bound directly with `ItemsSource="{Binding <Region>View}"`; no docking framework is involved. Author equivalent Avalonia `TabControl`s. Each region binds to the corresponding neutral tab-view exposed by the Avalonia panel VM (the per-location views: `CenterTopView`, `CenterBottomView`, `RightTopView`, `RightBottomView`, `LeftView`; `CenterView` is the obsolete `TabLocation.Center` and is unused by the live layout — wire only the five active regions). Each view is an `IFilteredTabView` (Task 6), **not** a WPF `ICollectionView`. Tab headers come from the tab model's title, or its custom-header content when present (the WPF head selects via `PluginTabHeaderTemplateSelector` — replicate with an Avalonia header `DataTemplate`/selector), with the tab's content handle (the neutral `IPanelContent` from Phase 5, not WPF `FrameworkElement`) as the body. For the Avalonia head, bodies are placeholder controls; binding the real `IPanelContent` to live Avalonia controls is a Phase 15/20 concern and is noted as a risk. Dock.Avalonia is not required to meet this phase (see Decisions & rationale).

6. **Neutral tab-view seam (replaces `ICollectionView`).** Today's WPF `MainPanelViewModel` exposes six `ICollectionView` properties (`System.Windows.Data`, WPF) — `CenterView`, `RightBottomView`, `RightTopView`, `CenterTopView`, `CenterBottomView`, `LeftView` — each built from a shared `ObservableCollection<PluginTab>` via `ListCollectionView` with `IsLiveFiltering = true` filtering on `PluginTab.IsVisible`/`Location`. Avalonia has no `ICollectionView`/`ListCollectionView`. Two paths were considered:
   - (a) Bind Avalonia tab `ItemsSource` directly to the `ICollectionView` (it implements `IEnumerable` + `INotifyCollectionChanged`) and drive selection via `CurrentItem`/`MoveCurrentTo`. **This only works if `ICollectionView` is reachable on a non-Windows TFM — and it is not** (it lives in a Windows-only WPF assembly), so direct-binding it from the Avalonia head pulls a WPF assembly into that head's reference graph, which the Phase 7 purity guard rejects. Path (a) is therefore **not viable**; do not plan on it.
   - (b) Introduce a neutral view-collection seam in `Gum.UiAbstractions` (e.g. `IFilteredTabView` exposing an observable, filtered, selection-tracking collection) that the shared neutral panel VM exposes instead of `ICollectionView`, with a WPF adapter over `ListCollectionView` (used by `MainPanelViewModel` in `AddGumWpf`) and an Avalonia-native impl. **This is the mandatory path** — it keeps the Avalonia head free of WPF references and is the only option that satisfies the purity guard.
   Implement (b). **This seam is part of Phase 5's panel/tab-host seam family** (it lives in `Gum.UiAbstractions` and reshapes the shared panel VM), so coordinate it back into Phase 5 rather than building a one-off here — Phase 5's panel/tab-host section names this ownership explicitly. Capture the concrete `IFilteredTabView` shape and the per-head impls in this phase's notes.

7. **Menus & toolbars in AXAML.** The WPF head builds its menu imperatively via `MenuStripManager` (registered in `Builder.cs`); the menu is not declared in `MainWindow.xaml`. For the skeleton, author a static Avalonia `Menu` (and any toolbars) in AXAML that invokes the command services (`IEditCommands`/`IFileCommands`/`IGuiCommands`/`IWireframeCommands`) directly — do **not** attempt to port `MenuStripManager` or reuse the WPF menu model in this phase. A head-agnostic menu model can be considered later (flag to architect). A small representative set of menu items (e.g. File → New/Open/Save, Edit → Undo/Redo) is sufficient to prove command-service binding.

8. **App lifecycle parity.** Replicate `Program.cs` startup/teardown:
   - On startup, resolve `IMessenger` and send `ApplicationStartupMessage`.
   - On exit/shutdown (`IClassicDesktopStyleApplicationLifetime.Exit` or `ShutdownRequested`), build the `teardownActions` list, send `ApplicationTeardownMessage(teardownActions)`, then invoke each action. The WPF `MainPanelViewModel` registers a teardown handler that persists layout via `SaveLayoutSettings`; the shared neutral base VM should carry that same `IRecipient<ApplicationTeardownMessage>` layout-persistence behavior so the Avalonia panel VM inherits it — confirm it fires under the Avalonia exit path.
   - Decide where the equivalent of `InitializeGum(...)` runs for the Avalonia head; much of it (TypeManager, PluginManager, WireframeObjectManager) is canvas/plugin-heavy and may be deferred to Phase 15. For the skeleton, run only the minimum needed to populate placeholder panels (e.g. `LoadSettings`, theme apply) and gate the rest.

## Risks & notes

- **ICollectionView / Avalonia collection-view gap (highest risk).** `ICollectionView`/`ListCollectionView` are `System.Windows.Data` (WPF), live in a Windows-only assembly, and have no Avalonia equivalent. The WPF `MainPanelViewModel` builds six of them with live filtering and current-item tracking. The direct-bind shortcut (Task 6 option (a)) **only works if `ICollectionView` is reachable on a non-Windows TFM — it is not** — so it would drag a WPF assembly into the Avalonia head and fail the Phase 7 purity guard. The mandatory resolution is the neutral filtered-view seam in `Gum.UiAbstractions` (`IFilteredTabView`) with per-head impls (WPF adapter over `ListCollectionView`, Avalonia-native impl). This modifies a shared VM and is owned by Phase 5's panel/tab-host seam family; coordinate it there. Decide the concrete shape early; this gates the whole panel layout.
- **`PluginTab.Content` is `System.Windows.FrameworkElement`.** The tab body type is WPF-specific. Phase 5 neutralizes the content handle to `IPanelContent`; for the skeleton, tab bodies are placeholders, so binding real `IPanelContent` to live Avalonia controls is deferred to Phase 15/20 (which will need a per-head content adapter). Flag now; do not attempt the live-content hosting in this phase.
- **`MainPanelViewModel` WPF/WinForms coupling.** It imports `System.Windows`, `System.Windows.Data`, and `System.Windows.Forms.Integration` (`WindowsFormsHost`, used by `AddControl(Control, ...)`). Because it stays the WPF `ITabManager` impl (registered only in `AddGumWpf`) and is **not** consumed by the Avalonia head, the Avalonia head does not transitively reference these WPF/WinForms types. The risk to watch is that the shared neutral base VM extracted from it stays clean of `System.Windows*`/`WindowsFormsHost` so the Phase 7 purity guard holds. Tied to the neutral tab-view seam decision.
- **Preserve the `SelectedItem`→`LeftView.CurrentItem` binding quirk deliberately (do NOT "fix" it).** `MainPanelControl.xaml:135` binds `RightBottomTabControl.SelectedItem` to `LeftView.CurrentItem` (`Mode=OneWay`) — almost certainly a copy-paste slip in the WPF shell (it cross-wires the bottom-right tab's selection to the *left* view's current item). The Avalonia re-author must **replicate this behavior as-is**, not silently correct it, so the port is behavior-preserving and the cutover stays diffable. The maintainers should file the suspected bug separately and decide on a fix in its own change, not fold a behavior change into the migration.
- **No docking framework in the current shell (premise correction).** A common assumption is that the shell uses Xceed AvalonDock and must be replaced by Dock.Avalonia. That is **not** the case: the verified WPF shell layout (`MainPanelControl.xaml`) is plain `TabControl`s in a `Grid`/`GridSplitter`. AvalonDock is referenced only for a couple of utility types in `Dialog.cs` and as assembly references in `Gum.csproj`. Therefore Phase 10 does **not** need Dock.Avalonia — plain Avalonia `TabControl` + `GridSplitter` reaches parity. Dock.Avalonia is a future option only if drag-rearrange/float/persisted dock layouts are wanted; if adopted later it brings its own layout model/serialization differences, but that is out of scope here.
- **Layout persistence via `LayoutSettings`.** Column/row sizes persist through `MainPanelViewModel.SaveLayoutSettings` (the VM registers this via `IRecipient<ApplicationTeardownMessage>.Receive` → `message.OnTearDown(SaveLayoutSettings)`), writing `MainTabDimensions` (LeftColumnWidth/CenterColumnWidth/BottomRightHeight) to `appsettings.json` via `IWritableOptions<LayoutSettings>`. This only fires if the Avalonia exit path actually sends `ApplicationTeardownMessage` **and runs the collected actions** (Task 8); otherwise sizes silently stop persisting. Window bounds persist via `WindowSettings` (Width/Height/Top/Left/IsMaximized) — these need equivalent two-way binding on the Avalonia window. The WPF `MainWindow` two-way-binds `Width`/`Height`/`Left`/`Top`/`WindowState` to `MainWindowViewModel`, which maps to/from `WindowSettings.IsMaximized`; replicate that mapping for the Avalonia window (note Avalonia uses `WindowState`/`Position` rather than `Left`/`Top`).
- **Custom chrome deferred.** The WPF window is a ControlzEx `WindowChromeWindow` with hand-built caption buttons and hit-test regions. Reproducing that in Avalonia is non-trivial and out of scope; use standard window decorations now and revisit in Phase 25.
- **`InitializeGum` ordering.** `Program.cs:InitializeGum` runs a long, order-sensitive init chain (settings → theme → `TypeManager` → tree view → wireframe → property grid → plugins → project load → file-watch timer). Most is canvas/plugin-dependent and should be deferred; running it wholesale before Phase 15 risks pulling in WinForms/XNA wireframe init that the Avalonia head can't yet host. Run only the subset needed for placeholder panels and gate the rest behind later phases.
